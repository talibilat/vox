## What was built

- `earshot/barge/vad.py`: `SpeechOnsetDetector`, Silero VAD (openWakeWord's ONNX build) over mic frames, scored as 640-sample sub-chunks with a 2-sub-chunk patience so an onset decision can complete within a single 80ms mic frame, plus a pre-roll ring buffer so the interrupting utterance's start is captured. Echo-discrimination assumptions (headset or modest speaker volume, no AEC) are documented in the module docstring; P3-01 later tuned the shipped threshold and recorded remaining live-room checks.
- `earshot/barge/interrupt.py`: `InterruptibleVoiceLoop`, the LISTENING / RECORDING / RESPONDING state machine that supersedes the plain InputPipeline wiring. During RESPONDING the mic feeds the onset detector; an onset (or the escape hatch) cancels synthesis, stops playback, and jumps straight to RECORDING with the pre-roll audio, so the interruption becomes the next command with no wake word. Every interrupt logs onset-to-silence latency and keeps `latencies_ms` for the distribution.
- `earshot/output.py`: `cancel_current()` added so the in-flight `speak_stream` stops consuming the agent stream and synthesizing when barged in.
- `earshot/audio/playback.py`: playback slice halved to ~23ms (it spends directly from the 200ms budget), and two real-hardware bugs fixed (below).
- `earshot/daemon.py` and `earshot/cli.py`: the daemon wires `InterruptibleVoiceLoop`; `earshot interrupt` sends SIGUSR1 as the push-to-interrupt escape hatch (bind it to a system hotkey for one-keystroke barge-in); the handler is registered before the voice loop exists and unconditionally.

## Real-hardware bugs found while measuring (fake sinks could not catch these)

1. `SounddeviceSink.abort()` reused the aborted CoreAudio stream; the next write failed with PaErrorCode -9986, the playback worker died, and every later stop hung for its 2s idle timeout (measured 2112ms interrupts). Fix: replace the stream on abort.
2. Deferring the replacement stream open to the playback thread raced it against the interrupter's close; CoreAudio intermittently failed with "Could not obtain stream info" or deadlocked. Fix: all sink operations serialized by a lock, so abort waits at most one ~23ms in-flight slice.
3. The playback worker now survives sink write failures instead of dying silently (a dead worker turns every stop into a 2s timeout).

## Latency accounting against the 200ms target

The instrumented number covers detection-to-silence (cancel, stop, queue flush, stream swap): ~124ms median.
Detection itself adds audio-time lag before that clock starts: bounded by the sub-chunk design at one to two mic frames (80 to 160ms), with the single-frame case the common one because the patience window fits inside one frame.
Median total is therefore ~200ms and the common case is well under; P3-01 revalidated the stop path after tuning, and smaller mic blocks remain the known lever if real-world use wants more margin.
